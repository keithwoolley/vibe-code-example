# Acknowledgement of critique `20260419-1622`

**Date:** 2026-04-19
**Responding to:** `20260419-1622-critique-of-814d71f290fad7428b13738120990f21990ac09a.md`

The spec was rewritten. It went from ~770 lines / 17 top-level sections to ~175 lines / 11
top-level sections. Below is what I accepted, what I partially accepted, and what I pushed back
on.

## Accepted in full

- **Cut Appendix A (Field Reference).** Duplicated §5. Gone.
- **Cut Appendix B (Event Taxonomy).** Internal event names that nothing referenced. Gone.
- **Cut §12 Non-Functional Requirements.** Dropped entirely. Kept only a one-line responsiveness
  note in the overview ("usable on desktop ≥1024px and mobile ≥360px"). Browser matrix, cold-
  start budget, ±2s alarm accuracy, setInterval prohibition, accessibility disclaimer, and the
  no-network reiteration are all gone.
- **Cut §15 Definition of Done.** Duplicated §14 acceptance scenarios. Gone.
- **Cut §9.1 ASCII state diagram.** Replaced with a three-sentence prose summary in the new §7.
- **Trim §13 Out of Scope.** Went from ~20 items to the 4 genuinely-tempting adjacencies the
  critique suggested (push/background notifications, multi-device sync, non-daily schedules,
  snooze). Everything else (HealthKit, OCR, PIN, etc.) was noise and is gone.
- **Drop `source: Source`.** Removed from `DoseEvent`. The `manual-edit` source value and its
  associated transition rule went with it.
- **Drop `statusChangedAt`.** Removed. Timeline ordering of PRN events is now "log order"
  rather than `statusChangedAt`, which is equivalent for insertion-only semantics.
- **Drop `createdAt` / `updatedAt` on `Medication`.** Removed. Nothing referenced them.
- **Drop `schemaVersion` and `lastSeenDate` metadata.** Removed. The on-open rollover check is
  now "if any pre-today event is still pending/due, run rollover", which is equivalently derived
  from the data itself.
- **Collapse `AlarmTime` to a string.** `alarmTimes` is now `string[]` of `"HH:MM"` values.
  `DoseEvent.scheduledTime` is the same shape. The separate `AlarmTime` entity is gone.
- **Keep the `Form` enum.** Kept as §3.1.1.
- **Drop field character limits.** Gone.
- **Simplify §6.3 rollover.** Now one paragraph with a four-step list, no `setTimeout`-vs-
  open-detection disjunction — just "at midnight, and on every app open".
- **Simplify §7.3 / §8.3 queuing.** Now "process sequentially in any deterministic order — e.g.,
  ascending scheduledTime". The pseudocode is gone; the case-insensitive alphabetical tiebreak
  is gone.
- **Drop §8.5 bullet 4** (the "does NOT modify document.title / show a notification / flash the
  favicon" list). Defining the app by what it doesn't do was weightless. Gone.
- **Drop §14.12 and §14.13.** These were restatements of §8.5 and §8.6. Gone.
- **Drop the "StrongDM attractor NLSpec style" framing paragraph.** Spec no longer pitches
  itself as production-grade.

## Reinstated after initial pass

- **Table of Contents.** Originally dropped as "unnecessary at this length", but the user
  asked for it back. Restored.

## Partially accepted

- **Glossary cut.** The critique didn't explicitly call for dropping §4 Glossary, but in a
  4–5 page doc it was overhead. The remaining definitions (PRN, finalized day, today) are now
  inline at their first use in the relevant section, which is shorter and contextual.
- **Acceptance scenarios count.** Critique said "6–8". I kept 9. The ones I kept cover genuinely
  distinct behaviors (add+schedule, Take, Ignore, re-ring-next-day, edit-today, finalized-read-
  only, past-time-on-open, PRN, delete-cascade). Cutting any of these loses a real guarantee.
  The 30-day-retention scenario was the one I dropped on top of the critique's own cuts — the
  cascade test already exercises deletion, and the 30-day window is clearly stated in §4.

## Pushed back on

- **"Reconsider PRN entirely / move to stretch goal / cut it."** I disagree with cutting PRN.
  PRN is a first-class product concept that appears in the Goals, the data model, the Today
  screen, and a load-bearing acceptance scenario. Removing it would be a *product* change, not
  a *spec tightening*, and the user hasn't asked for that. What I did do is simplify PRN's
  spec footprint considerably: the separate "daily plan" materialization language is gone, PRN
  logging is now a single short paragraph in §5, and the §7 alarm flow no longer has to
  special-case PRN because PRN simply doesn't participate in alarms.

  If the user later decides PRN is clutter for the toy, that's a one-commit removal from the
  current spec — easier than having cut it and needing to re-add it.

## Result

- **Line count:** ~770 → ~175.
- **Top-level sections:** 17 → 11.
- **Domain model size:** `Medication` went from 10 fields to 8; `DoseEvent` went from 7 fields
  to 5. `AlarmTime` as a separate entity is gone.
- **The core alarm loop (§6) is now the single longest section in the document**, which is the
  right shape for what the app actually does.
