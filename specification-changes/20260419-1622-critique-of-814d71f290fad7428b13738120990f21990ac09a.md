# Critique of `specification.md` @ 814d71f

**Date:** 2026-04-19 16:22
**Spec SHA:** 814d71f290fad7428b13738120990f21990ac09a
**Context:** This is a vibe-coding example app. The spec should give a coding agent enough to build the right thing in one shot — not stand up to a compliance audit. The current spec reads like a production-grade NLSpec for a shipping product. For a toy, most of it is overhead.

---

## Overall take

The spec is ~770 lines and 17 top-level sections for what is, functionally, a single-page app with four screens and one interesting interaction (alarm → Take/Ignore). A coding agent will build this correctly from roughly a third of the current content. Most of the rest is ceremony: restated fields, state-machine formalism, explicit non-goals, and operational rules that add no new information once the happy path is specified.

The good news: the **core** is well-chosen. "The alarm response *is* the log" is a strong product thesis, and §§1–2, §7, §8 communicate it cleanly. The critique below is about removing weight, not rewriting the concept.

---

## What to cut outright

These sections are pure overhead for a toy example.

1. **Appendix A — Field Reference.** Restates §5 in table form. Pick one or the other; §5 is more readable.
2. **Appendix B — Event Taxonomy.** Defines internal runtime events that no part of the UI spec depends on. This is framework scaffolding for a system that doesn't exist yet. Delete.
3. **§12 Non-Functional Requirements.** For a vibe-coding toy:
   - 12.1 (responsive to 360px): yes, keep as one sentence in §1.
   - 12.2 (browser matrix): nobody is QA-ing this on four browsers. Drop.
   - 12.3 (cold-start < 1s, ±2s alarm accuracy, no `setInterval`): implementation micromanagement. Drop.
   - 12.4 (accessibility not a priority): already implied. Drop.
   - 12.5 (no network): already in §11.3. Drop.
4. **§13 Out of Scope.** Useful in principle, but the current list is 20+ items long and reads defensively. Keep 3–4 genuinely tempting adjacencies (push notifications, multi-device sync, non-daily schedules) and delete the rest — nobody is going to speculatively add HealthKit integration to a toy.
5. **§15 Definition of Done.** 17 checklist items that duplicate §14 acceptance scenarios. Pick one format. §14 is more useful.
6. **§9.1 ASCII state diagram.** The prose in §8 and §9.2 already covers it. The diagram adds nothing a reader needs.

Cutting the above removes ~40% of the document with zero loss of implementable intent.

## What to simplify in the domain model

§5 is the most over-engineered part of the spec for a toy.

1. **Drop `source: Source`.** Tracking whether a dose event came from `scheduled` / `prn` / `manual-edit` is audit-log thinking. The UI never surfaces it. If you need to distinguish "is this a PRN event?" you already know from `scheduledTime == null`.
2. **Drop `statusChangedAt`.** Nothing in the UI depends on it. If you later want "last changed at" in history, add it then.
3. **Drop `createdAt` / `updatedAt` on `Medication`.** No screen shows them. No rule depends on them.
4. **Drop `schemaVersion` and the `lastSeenDate` metadata record (§11.2).** Schema migrations on a toy are YAGNI. `lastSeenDate` is used only for the rollover-on-reopen fallback, which can equivalently be derived by comparing the max `date` on any existing `DoseEvent` to today.
5. **Collapse `AlarmTime` to a string.** `{hour: int, minute: int}` as a separate entity is over-modelling. Store `"HH:MM"` strings. Parse at alarm-scheduling time.
6. **Keep the `Form` enum.** It gives the user a recognizable list of common medication types in the dropdown, which is worth the small amount of surface area.
7. **Drop field character limits** (name ≤ 120, notes ≤ 1000). Not enforced anywhere that matters for a local-only toy.

Result: `Medication` becomes ~4 fields (id, name, isPRN, alarmTimes) plus a couple of optional display strings. `DoseEvent` becomes ~4 fields (id, medicationId, date, scheduledTime, status).

## What to simplify in behavior

1. **§7.3 / §8.3 past-time and sequential alarm queuing.** The pseudocode and ordering rules (ascending `scheduledTime`, case-insensitive alphabetical tiebreak) are more specification than the problem needs. For a toy: "if multiple alarms are due, show them one at a time in any deterministic order." That's it.
2. **§6.3 day rollover as a formally ordered 4-step procedure with a `setTimeout`-vs-open-detection trigger disjunction.** Simplify to: "At midnight, and on every app open, mark any pre-today `pending`/`due` events as `ignored`, create today's scheduled events if missing, and delete events older than 30 days." One paragraph.
3. **§8.5 alarm sound**, bullet 4 ("the app does NOT modify `document.title`, show a notification, flash the favicon..."). This is defining the app by what it *doesn't* do in a way no one would have built anyway. Drop.
4. **§9.2 `source = manual-edit` on flips.** See domain-model point 1 — if `source` goes away, this goes with it.
5. **Reconsider PRN entirely.** PRN doubles the surface area of the spec (separate button, separate logging flow, null `scheduledTime`, special-cased event ordering). If the vibe-coding goal is "show the alarm → Take/Ignore loop," PRN is a distraction. Consider moving it to a "stretch goal" list or cutting it.

## What to keep

To be clear, the following are pulling their weight and should stay:

- §1 Overview and §2.1 Goals — concise and well-scoped.
- §3 Design Principles — these genuinely guide tradeoffs (especially "immutability of the past" and "prescriptive defaults").
- §5.1 `Medication` and §5.3 `DoseEvent` *after* the cuts above.
- §6.2 definition of "today" and §6.4 30-day retention.
- §7.1 daily-only schedule thesis.
- §8.1, §8.2, §8.4 alarm flow — this is the core and it's clearly written.
- §10.2 Today screen (timeline + missed-today section) — the product shape.
- §10.4.1 deletion cascade.
- §14 acceptance scenarios — 6–8 of the most load-bearing ones. Drop §14.12 and §14.13 (they specify behavior that's obvious from §8.5/§8.6).

## Suggested target shape

A tighter version would be roughly:

1. Overview (½ page)
2. Goals (3–4 bullets)
3. Domain model: Medication, DoseEvent (½ page total, no appendix)
4. Day boundary + rollover (½ page)
5. Alarm flow: when it fires, the modal, Take vs Ignore (1 page — the heart of the doc)
6. Screens: Today, Medications, History (1 page)
7. Persistence: "IndexedDB or localStorage, your call, no network" (3 sentences)
8. Acceptance scenarios: 6–8 of them (1 page)

Roughly 4–5 pages total instead of ~20. A coding agent will produce the same app from the tighter spec, faster, with fewer places to get distracted.

---

## Summary

The spec is well-written for what it is — but what it is is a production NLSpec, not a vibe-coding brief. The cuts above remove ceremony (audit fields, schema versions, formal queuing rules, a redundant appendix, a redundant DoD, and an exhaustive out-of-scope list) without weakening the product definition. The remaining document would be tighter, faster to read, and unambiguous in the places that actually matter: the alarm loop and the immutability-at-midnight rule.
